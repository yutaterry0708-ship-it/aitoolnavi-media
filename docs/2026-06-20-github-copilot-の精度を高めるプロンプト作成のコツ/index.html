GitHub Copilotを使っているのに、なぜか提案がズレる──その原因はプロンプトにあった

「GitHub Copilotを導入したのに、思ったより使えない」「補完が的外れで、結局自分で全部書いている」──こういった声は、開発者コミュニティでよく耳にします。私自身も最初の数週間はまったく同じ状態でした。

結論から言うと、GitHub Copilotの精度はプロンプト（コメントや周辺コード）の書き方で大きく変わります。ツール側の問題ではなく、"どう伝えるか"の問題であることがほとんどです。

この記事では、実際にCopilotを日常的に使い込んできた経験をもとに、精度を引き上げるための具体的なプロンプト設計のコツを体系的に解説します。

GitHub Copilotの精度が低く感じられる根本的な理由

Copilotは「文脈を読む」AIである

GitHub Copilotは、OpenAIのCodexをベースにした大規模言語モデルです。重要なのは、Copilotは「命令を実行する」というより「文脈から次のコードを予測する」という動作原理を持っている点です。

つまり、周辺のコード・コメント・ファイル名・関数名などすべてが"プロンプト"として機能しています。ここを理解していないと、どれだけ使い続けても精度は上がりません。

よくある精度低下の原因3つ

  1. コメントが曖昧すぎる（例：// データを処理する）
  2. コンテキストが少ない（新規ファイルで何も書かずに補完を期待する）
  3. 変数名・関数名が意味を持っていない（a, tmp, func1 など）

私の運用では、この3つを意識するだけで提案の質が体感で大きく変わりました。

精度を高めるプロンプト作成のコツ【実践編】

コツ1：コメントは「目的 + 入力 + 出力」の3点セットで書く

最もシンプルかつ効果的な方法です。

悪い例：

def process(data):

良い例：

# ユーザーの購入履歴リスト（辞書型のリスト）を受け取り、
# 合計金額と平均金額を計算して辞書で返す関数
def calculate_purchase_summary(purchase_history: list[dict]) -> dict:

後者のコメントでは、Copilotが「何を受け取って何を返すか」を明確に把握できるため、的外れな補完が激減します。実際に試してみると、型ヒントとコメントの組み合わせが特に効果的でした。

コツ2：関数名・変数名に意味を持たせる

Copilotはコード全体を文脈として読みます。get_data() より fetch_active_users_from_db() のほうが、補完の精度が上がります。

ポイント：
- 動詞＋対象＋修飾語の命名規則を徹底する
- 略語は避ける（usr より user、calc より calculate）
- 型ヒント（TypeScript/Pythonなら型アノテーション）を積極的に使う

これは「Copilotのため」だけでなく、コードの可読性向上にもつながるため、チーム開発での副次的なメリットも大きいです。

コツ3：既存コードをコンテキストとして活用する

Copilotは現在開いているファイルと、隣接するタブのファイルを参照します。精度を上げるには、関連コードを同じファイルか近くのタブに開いておくことが重要です。

私の運用では、以下のような工夫をしています：

  • 似たパターンの関数を同じファイルに書いておき、それを"見本"として使わせる
  • インターフェース定義やスキーマファイルを別タブで開いた状態にする
  • テストファイルを先に書いて、実装コードをCopilotに補完させる（TDDとの相性が良い）

特に「既存の類似関数の直下に新しい関数を書く」という方法は、スタイルの一貫性も保てるため非常に実用的です。

コツ4：Copilot Chatでは「役割」を明示する

GitHub Copilot Chatを使う場合、プロンプトの冒頭に役割を与えると出力品質が上がります。

例：

あなたはPythonのシニアエンジニアです。
以下のコードをレビューして、パフォーマンス上の問題点を3つ指摘してください。
コードの可読性については今回は無視してください。

[コードを貼り付け]

「範囲を絞る」「不要な観点を除外する」という指示が、回答の的確さを大きく高めます。これはChatGPTなど他のLLMツールにも共通するテクニックですが、Copilot Chatでも同様に有効です。

コツ5：.github/copilot-instructions.md でプロジェクト固有の文脈を設定する

2024年後半から利用可能になった機能で、リポジトリルートに copilot-instructions.md を置くことで、プロジェクト全体のコーディング規約やアーキテクチャ方針をCopilotに事前に伝えられます。

記載例：

## プロジェクト概要
このリポジトリはFastAPIを使ったREST APIサーバーです。

## コーディング規約
- 型ヒントを必ず付ける
- 例外処理はカスタム例外クラスを使う
- テストはpytestで書く

## 禁止事項
- グローバル変数の使用
- `print()` によるデバッグ出力（loggingを使うこと）

この設定を入れてからは、補完コードがプロジェクトのスタイルに沿ったものになり、修正コストが明らかに減りました。チーム開発では特に効果を実感しやすい機能です。

正直なメリット・デメリット

✅ メリット

  • コメントを書く習慣が自然につく：プロンプトを意識することで、ドキュメントとしてのコメントも充実する
  • 型ヒントの活用が促進される：型を書くことでCopilotの精度が上がるため、型安全なコードが増える
  • TDDとの相性が抜群：テストを先に書いてから実装を補完させると、仕様に沿ったコードが生成されやすい
  • copilot-instructions.md による一括設定：チーム全体で恩恵を受けられる

❌ デメリット・注意点

  • 精度向上には学習コストがある：コツを身につけるまでに一定の試行錯誤が必要
  • 複雑なビジネスロジックには限界がある：ドメイン固有の複雑な仕様はコメントだけでは伝えきれないことも
  • 生成コードのレビューは必須：精度が上がっても、セキュリティ上の問題や微妙なバグが含まれることがある。盲目的に採用するのは危険
  • コンテキストウィンドウに上限がある：ファイルが大きすぎると参照できる範囲が制限される
  • プランによって機能差がある：copilot-instructions.md などの高度な機能はプランによって利用可否が異なる場合がある

よくある質問（FAQ）

Q1. GitHub Copilot IndividualとBusinessで精度に差はありますか？

A. モデル自体の精度差はほぼありませんが、BusinessプランではOrganization単位でのポリシー設定や、copilot-instructions.md の活用がしやすくなります。個人利用であればIndividualで十分ですが、チーム開発ではBusinessプランの管理機能が精度向上の運用面で有利です。

Q2. プロンプトを工夫しても提案が改善しない場合はどうすればよいですか？

A. まず「Copilot Chatに直接質問する」方法に切り替えてみてください。インライン補完が苦手なケースでも、Chatでは詳細な指示を与えられるため精度が上がることがあります。また、複数の候補を表示する Alt + ] / Option + ] のショートカットで別の提案を確認するのも有効です。それでも改善しない場合は、そのタスク自体がCopilotの得意領域外である可能性があります。

Q3. Copilotに書かせたコードはそのまま使っても大丈夫ですか？

A. 必ずレビューしてから使用してください。精度が上がっても、Copilotが生成するコードにはバグ・非効率な実装・セキュリティ上の問題が含まれることがあります。特に認証・暗号化・SQL生成などのセキュリティに関わるコードは、専門的な知識を持つ開発者が必ずチェックする運用を推奨します。

まとめ：GitHub Copilotの精度はプロンプト設計で変わる

GitHub Copilotの精度が上がらない最大の原因は、多くの場合「伝え方の問題」です。

この記事で紹介したコツをまとめると：

  1. コメントに「目的・入力・出力」を書く
  2. 意味のある命名規則を徹底する
  3. 関連ファイルをタブで開いてコンテキストを与える
  4. Copilot Chatでは役割と範囲を明示する
  5. copilot-instructions.md でプロジェクト固有の文脈を設定する

これらは特別な技術を必要とせず、今日から実践できるものばかりです。私の運用でも、この5つを意識するようになってから、補完コードの採用率が大きく改善しました。

ただし、どれだけ精度を上げても「生成コードのレビューを省略する」ことは避けてください。Copilotはあくまで強力なアシスタントであり、最終的な判断と責任は開発者自身にあります。
