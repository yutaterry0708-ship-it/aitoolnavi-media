## 「Copilot って思ったより使えない」と感じたことはありませんか？

GitHub Copilot を導入したものの、「なんか的外れな補完ばかり出てくる」「コメントを書いても意図が伝わらない」と感じて、結局手書きに戻してしまう——そんな経験をした開発者は少なくないはずです。

私自身、AIツールナビの編集部で複数のAIコーディングアシスタントを実際に使い込む中で、最初は同じ壁にぶつかりました。Copilot は使い方次第で補完精度が大きく変わるツールです。**「プロンプト（コンテキスト）の与え方」を変えるだけで、体感できるレベルで提案品質が上がります。**

この記事では、実際の運用で効果を感じたプロンプト作成のコツを体系的にまとめます。設定面の見直しから、コメントの書き方、Copilot Chat との使い分けまで、すぐ実践できる内容に絞って解説します。

## GitHub Copilot の精度が低く感じる根本的な原因

### Copilot はコンテキストを読んで補完する

GitHub Copilot の補完エンジンは、カーソル位置の**前後のコード・コメント・ファイル名・関連ファイル**などをコンテキストとして取り込み、最も確率の高いコードを予測します。つまり、**コンテキストが貧弱だと提案も貧弱になる**のは構造上避けられません。

「使えない」と感じる場面の多くは、Copilot のモデルが悪いのではなく、**モデルに渡している情報が不十分**なケースです。実際に使ってみて痛感したのですが、ファイルを新規作成してすぐ関数を書こうとすると、汎用的すぎる補完しか出てきません。一方、同じ関数でも周辺に型定義・既存関数・詳細なコメントがあると、驚くほどピンポイントな提案が返ってきます。

### 精度低下の主な原因まとめ

| 原因 | 具体的な症状 |
|---|---|
| コンテキスト不足 | 汎用的すぎる・関係ない補完が出る |
| コメントが曖昧 | 意図と異なる実装が提案される |
| ファイルが大きすぎる | コンテキストウィンドウから重要部分が溢れる |
| 言語・フレームワーク情報がない | 別スタックのコードが混入する |
| Copilot の設定が未調整 | 全言語で同じ挙動になる |

## 精度を高めるプロンプト作成のコツ【実践編】

### コツ1：コメントは「目的 → 入力 → 出力 → 制約」の順で書く

Copilot への最大のプロンプトは**インラインコメント**です。「// ユーザー一覧を取得する」のような一行コメントより、以下のように構造化すると精度が上がります。

```python
# 目的: ユーザーIDのリストを受け取り、アクティブなユーザーのみを返す
# 入力: user_ids (List[int])
# 出力: List[User] - is_active=True のユーザーのみ
# 制約: DB へのクエリは1回に抑えること。削除済みユーザーは除外する
def get_active_users(user_ids: list[int]) -> list[User]:
```

私の運用では、このフォーマットをスニペットとして登録しておき、複雑な関数を書くときはテンプレとして貼り付けています。コメントを埋めるだけで Copilot の提案精度が明確に変わります。

### コツ2：型アノテーションを必ず書く

型情報はコメントと同等かそれ以上に強力なヒントになります。特に TypeScript・Python・Rust などの型システムが充実した言語では、**型を書くだけで補完が劇的に絞られます。**

```typescript
// 型なし → 汎用的な補完
function processOrder(order) {

// 型あり → 具体的な補完
function processOrder(order: Order): Promise<ProcessedOrder> {
```

実際に使ってみて、TypeScript プロジェクトで `any` を多用していた頃と、型を厳密に書くようにしてからとでは、補完の的中率が体感で大きく異なりました。

### コツ3：関連ファイルを開いたままにする

Copilot（特に最新バージョン）は、**エディタで開いているタブのコード**もコンテキストとして参照します。これを意識的に活用します。

- 実装する前に、関連する型定義ファイル（`types.ts`、`models.py` など）を開いておく
- 参考にしてほしいユーティリティ関数のファイルを開いておく
- テストファイルを開いておくと、テストしやすい実装を提案しやすくなる

逆に言えば、**不要なファイルを閉じてコンテキストを整理する**ことも精度向上につながります。

### コツ4：ファイル冒頭に「コンテキストブロック」を置く

新規ファイルを作成するとき、冒頭に以下のようなブロックを置くと、ファイル全体の補完品質が上がります。

```python
"""
モジュール: 注文処理サービス
フレームワーク: FastAPI + SQLAlchemy
DB: PostgreSQL
規約: 非同期処理は async/await を使用。例外は OrderError を継承すること
"""
```

これは特にチーム開発で有効で、コーディング規約をコメントとして埋め込むことで、Copilot がプロジェクト固有のスタイルに寄せた提案をしやすくなります。

### コツ5：Copilot Chat を「設計の壁打ち」に使う

インライン補完だけでなく、**Copilot Chat（`/explain`・`/fix`・`/tests` コマンド）**を組み合わせると精度向上の幅が広がります。

私の運用では以下のフローが定着しています。

1. **Chat で設計を固める**：「この関数の責務をどう分割すべきか」を Chat に相談
2. **コメントを Chat で生成**：「この関数の JSDoc を書いて」と依頼してコメントを充実させる
3. **インライン補完で実装**：充実したコメント・型を元に Copilot に補完させる
4. **`/tests` でテスト生成**：実装後にテストケースを自動生成

このフローにしてから、補完の手戻りが明らかに減りました。

## 正直なメリット・デメリット

### メリット

- **コンテキストの与え方を工夫するだけで精度が上がる**：追加費用なし、設定変更なしで効果を感じやすい
- **`copilot-instructions.md` でプロジェクト全体に規約を適用できる**：チーム開発で特に有効
- **Chat コマンドが充実している**：`/fix`・`/explain`・`/tests` など目的別に使い分けられる
- **主要IDE との統合が安定している**：JetBrains 系でも動作が安定しており、乗り換えコストが低い

### デメリット（正直に書きます）

- **コンテキストウィンドウに限界がある**：大規模なモノリポでは、関連ファイルが多すぎてコンテキストが溢れる場面がある
- **プロンプトエンジニアリングの学習コストがかかる**：「入れるだけで使える」と思って導入すると期待外れになりやすい
- **補完の採用・却下の繰り返しが疲弊感につながることも**：提案の質が低い言語（ニッチなDSLなど）では逆に邪魔に感じる
- **セキュリティポリシーによっては導入できない環境がある**：コードがサーバーに送信されるため、機密性の高いプロジェクトでは要確認
- **Cursor など競合と比較するとリポジトリ全体把握は弱い**：プロジェクト横断の補完精度では差を感じる場面がある

## よくある質問（FAQ）

### Q1. `copilot-instructions.md` とは何ですか？効果はありますか？

**A.** リポジトリのルートまたは `.github/` ディレクトリに置くことで、**プロジェクト全体の Copilot の挙動をカスタマイズできるファイル**です。コーディング規約・使用フレームワーク・禁止パターンなどを記述すると、補完・Chat の両方に反映されます。実際に使ってみて、チーム開発での規約遵守率が上がる効果を感じました。ただし VS Code での対応が先行しており、JetBrains 系では機能が限定的な場合があります。最新の対応状況は公式ドキュメントで確認してください。

### Q2. コメントを英語で書くと精度が上がりますか？

**A.** モデルの学習データは英語が多いため、**英語コメントの方が補完精度が高い傾向はあります**。ただし日本語でも十分な精度が出るケースが多く、「日本語だから使えない」というほどの差ではありません。私の運用では、関数レベルのコメントは英語、ファイル全体の説明は日本語という使い分けをしています。チームの読みやすさと精度のバランスで判断するのが現実的です。

### Q3. Copilot Individual と Copilot Business で精度に差はありますか？

**A.** 補完モデル自体の精度に大きな差はありませんが、**Business・Enterprise プランでは組織単位のポリシー設定・`copilot-instructions.md` の組織適用・監査ログ**などが使えます。精度向上という観点では、Enterprise の「ナレッジベース（社内ドキュメントをコンテキストに追加）」機能が効果的です。個人開発なら Individual で十分ですが、チーム開発では Business 以上のプランを検討する価値があります。

## まとめ：Copilot の精度はプロンプト次第で変わる

GitHub Copilot の補完精度が低く感じる原因のほとんどは、**モデルへのコンテキスト（情報）の渡し方**にあります。

今日から実践できるポイントを整理します。

1. **コメントは「目的・入力・出力・制約」の構造で書く**
2. **型アノテーションを丁寧に書く**
3. **関連ファイルをエディタで開いておく**
4. **ファイル冒頭にコンテキストブロックを置く**
5. **Copilot Chat を設計・コメント生成に活用する**
6. **`copilot-instructions.md` でプロジェクト規約を明示する**

どれも追加費用なし・設定変更なしで今すぐ試せます。「Copilot を入れたけど使えない」と感じているなら、まずコメントの書き方を変えるところから始めてみてください。
